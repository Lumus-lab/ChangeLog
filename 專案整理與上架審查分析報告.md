## 一、送審前必修的阻塞項目

下面這些不是「可整理可不整理」，而是**不處理就不該送審**。

### 1. Android release 仍使用 debug signing，Google Play 不能這樣上架

證據：

- `android/app/build.gradle.kts:33-37`

現況：

- `release` build type 直接使用 `signingConfigs.getByName("debug")`
- 註解也明寫只是為了讓 `flutter run --release` 可以工作

風險：

- 這不是正式發佈配置
- 即使能打包，也不是應該提交到 Google Play 的發佈方式

建議做法：

- 建立正式 release keystore / Play App Signing 流程
- 把簽章設定移出 debug key
- 將簽章資訊改由 `key.properties` 或 CI secret 注入，不要寫死在 repo

### 2. 廣告在 release mode 仍使用 Google 測試廣告 ID

證據：

- `lib/services/ad_service.dart:7-30`

現況：

- `rewardedAdUnitId` 與 `interstitialAdUnitId` 在 `kReleaseMode` 下仍回傳 Google 官方 test ad unit
- 註解也寫了 `TODO: Replace with your actual production Ad Unit ID`

風險：

- 這代表 monetization 尚未完成
- Google 官方文件明確說測試用 ad unit ID 只能開發測試用，發佈前要替換成自己的正式 ad unit

建議做法：

- 建立正式 Android / iOS ad unit
- 測試、staging、production 分開配置
- 不要在 `AdService` 內用同一段 fallback 混用測試與正式邏輯

### 3. App 實際 bundle ID / application ID 與 Worker 驗證 ID 不一致

證據：

- `android/app/build.gradle.kts:24`
- `ios/Runner.xcodeproj/project.pbxproj:498`
- `ios/Runner.xcodeproj/project.pbxproj:680`
- `lib/services/ai_interpreter_service.dart:22`
- `worker/wrangler.toml:13`

現況：

- Android / iOS 都是 `com.lumus.changelog`
- Flutter app 傳給 Worker 的 `app-id` 是 `com.lumuslab.changelog`
- Worker 驗證的 `EXPECTED_APP_ID` 也是 `com.lumuslab.changelog`

白話解釋：

- 你可以把這個 ID 想成 App 的「身分證字號」或「員工證號」
- App 去找 Worker 幫忙做 AI 解卦時，Worker 會先檢查「你是不是我認得的那個 App」
- 但現在手機 App 自己的正式身分是 `com.lumus.changelog`
- 可是它拿去跟 Worker 報到時，卻報成 `com.lumuslab.changelog`
- 這就像同一個人，身分證寫 A，公司門禁刷卡卻刷成 B，門禁系統可能直接不讓你進
- 所以這不是抽象的技術潔癖，而是很可能讓 AI 功能直接失敗的原因

風險：

- 這會直接讓預設 AI 流程在正式環境可能 403 失敗
- 審查時如果 reviewer 碰到 AI 主流程壞掉，很容易落入 Apple 2.1 App Completeness 問題

建議做法：

- 先決定唯一正式 bundle ID / application ID
- App、Worker、文件、商店後台全部統一
- 把 app ID 變成單一來源，不要在多個檔案各自硬編碼
- 最簡單的方向是：選一個你要正式上架用的 ID，然後把 Android、iOS、Flutter 傳給 Worker 的值、Worker 期待的值，全都改成同一個

### 4. 隱私政策不符合兩大商店要求的形態與內容

證據：

- `Privacy.md` 只有 repo 內文字，App 內沒有可直接打開的完整隱私政策頁面或公開 URL
- `lib/views/settings_screen.dart:61-149` 只有簡短說明，沒有完整 privacy policy
- `lib/views/explanation_screen.dart:50-95` 也沒有隱私政策入口

官方要求重點：

- Apple 要求 App Store Connect metadata 要有 privacy policy 連結，而且 app 內也要容易存取；且政策必須清楚說明收集哪些資料、怎麼收、怎麼用、第三方共享、保存/刪除政策
- Google Play 要求所有 app 都要提供 privacy policy，且要是公開、可存取、非 PDF、非地區封鎖 URL；Data safety 內容也必須和實際行為一致

風險：

- 目前 repo 內的 `Privacy.md` 不等於商店可接受的 privacy policy 落地方式
- 內容也還不夠完整，缺少聯絡方式、保留/刪除政策、第三方 SDK 揭露完整度

建議做法：

- 準備一份正式公開網址的隱私政策頁面
- App 內增加「隱私政策」入口，至少在設定頁與首次說明頁都能找到
- 政策內容補齊：資料種類、傳輸對象、保存時間、刪除方式、第三方 SDK、聯絡窗口

### 5. 文件宣稱與實際實作不一致，容易觸發「metadata inaccurate / misleading」

證據：

- `README.md:16` 寫 `encrypted cloud backup`
- `Privacy.md:27-28` 又寫資料都只存在本機，除非主動分享匯出
- `lib/services/data_transfer_service.dart:11-52` 實作其實只是把 JSON 寫到 temp file 然後分享出去
- `lib/services/data_transfer_service.dart:56-101` 匯入則是選 JSON 檔重新寫回 DB

現況：

- 目前不是「完全沒有備份能力」
- 目前有的是「手動匯出 JSON 檔，再分享到 Google Drive 或其他地方，再手動匯入」
- 這種做法比較接近「手動備份 / 手動還原」
- 它不是內建雲端同步，也不是自動雲端備份
- 從 repo 內也看不到加密雲端備份流程、帳號綁定、同步狀態、版本控管、還原驗證

風險：

- Apple 2.1 / 2.3 明確要求 app 與 metadata 必須完整且準確
- Google Play 也要求 store listing 與資料揭露要與實際行為一致

白話解釋：

- 你的 app 現在確實可以把紀錄分享出去，分享到 Google Drive 也算成立
- 但這個能力的本質是「把檔案丟出去」，不是「app 自己有雲端備份系統」
- `README` 裡的 `encrypted cloud backup` 會讓人以為：
- app 有內建雲端帳號或備份機制
- 會自動備份或至少是 app 管理的雲端備份
- 備份內容有加密
- 這三件事，從目前程式看不到
- 所以問題不是「你完全不能備份」，而是「文案把手動匯出說得像真正的雲端備份」
- 另外，你提到現在 app 裡沒有「刪除占卦紀錄」功能，這會再延伸出另一個問題：如果之後要在隱私政策裡寫使用者可自行管理或刪除資料，實作也要跟得上

建議做法：

- 二選一：
- 要嘛把 README / 對外文案改成「本機匯出 / 匯入備份，可手動分享至 Google Drive 等雲端硬碟」
- 要嘛真正實作加密雲端備份，再補上隱私與資料刪除流程
- 如果近期不打算做完整雲端備份，我會建議直接把文案說成「手動備份」，這樣最準確

### 6. Worker 與資料處理的真實行為，現在不足以支撐目前的隱私說法

證據：

- `lib/services/ai_interpreter_service.dart:140-153`
- `worker/src/index.js:26-36`
- `worker/src/index.js:96-99`
- `Privacy.md:19-23`
- `Security.md:25-28`

現況：

- App 會把使用者問題、卦名、指引送到 Worker
- Worker 以 `question` 內容直接組 KV cache key
- `adToken` 目前只是 `'placeholder-token'`
- Worker 只檢查 header `app-id` 與 token 是否存在，不是真正驗證廣告獎勵
- `Access-Control-Allow-Origin` 是 `*`

白話解釋：

- 你可以把 Worker 想成一個中間轉運站
- 預設模式下，使用者在 app 裡輸入的問題，不是只留在手機裡
- 它會先送到你的 Worker，再由 Worker 送去 Gemini
- 也就是說，若使用的是預設 AI 模式，使用者的問題內容會離開手機
- 更重要的是，Worker 現在還會拿「問題文字」去組成快取 key
- 這表示系統不只是幫忙轉送，還把問題內容拿來參與後端儲存邏輯
- 所以如果對外文案寫得太像「資料都只在本機」「完全匿名、幾乎不留痕跡」，就會和實際行為有落差
- 另外，`adToken` 現在只是佔位字串，等於目前看不出來有真正做「你真的看完廣告才給額度」的後端驗證
- 這不一定會直接導致商店退件，但會讓隱私說明和安全說明顯得比實際做法更強

風險：

- 使用者問題本身可能非常敏感，現在不只是「匿名」，而是實際被送到第三方處理與快取
- 快取 key 直接使用使用者問題文字，是明顯的敏感資料處理設計問題
- 安全文件寫了 rate limit、驗證、防護，但 repo 內可見實作沒有達到文件描述程度

建議做法：

- cache key 不要直接用原始 question，至少要做不可逆 hash 或重新設計 caching 條件
- 補真正的 rewarded ad server-side verification 或至少可驗證 token 機制
- 如果 Worker 是正式產品一部分，隱私政策要明講第三方處理與保留策略
- 安全文案要回到實際可證明的程度，不要超額聲稱
- 若你想保留「隱私優先」定位，比較穩妥的寫法是：
- BYOK 模式：問題直接從使用者裝置送到 Google
- 預設模式：問題會經過你的 Cloudflare Worker，再轉送到 Google；系統可能為了成本與速度做快取
- 這樣就和實際設計一致了

### 7. BYOK 與 Worker 使用的 Gemini 模型不一致，而且 BYOK 目前不是你以為的 3.1

證據：

- `lib/services/ai_interpreter_service.dart:77`

判斷：

- 這點我前一版寫得不夠精準，這裡修正
- 你說「明明叫 Gemini 設定使用 `gemini-3.1-flash-lite-preview`」，這句話對了一半
- `worker/src/index.js:50` 的 Worker 路徑，確實是 `gemini-3.1-flash-lite-preview`
- 但 `lib/services/ai_interpreter_service.dart:77` 的 BYOK 直連路徑，實際寫的是 `gemini-1.5-flash`
- 也就是說：
- 沒填自己的 API Key 時，走 Worker，用的是 3.1
- 填了自己的 API Key 時，走 BYOK，反而改用 1.5
- 所以問題不只是「模型可能舊」，更直接的問題是「兩條路徑用的不是同一個模型」

白話解釋：

- 你可以把它想成同一家店有兩個櫃台
- 一個櫃台（Worker）賣的是新版產品 3.1
- 另一個櫃台（BYOK）賣的卻是舊版產品 1.5
- 使用者表面上看到的是同一個功能「AI 解卦」
- 但實際上是否有填自己的 API Key，背後用的模型就不同
- 這會造成結果風格、品質、速度、可用性都可能不一致
- 所以你現在的理解沒有錯：你以為整個系統都設定成 3.1，但從程式來看，只有 Worker 那條路是 3.1

風險：

- BYOK 是你產品的重要賣點之一
- 如果 reviewer 或使用者輸入自己的 key 後，得到和預設模式差很多的結果，甚至失效，會直接影響完整度與信任

建議做法：

- 先決定「預設模式」與「BYOK 模式」是否應該使用同一個模型
- 如果你的產品設計本意就是都用 3.1，那 BYOK 路徑目前明顯沒有跟上
- 把模型名稱抽成可配置值，不要散落在程式內
- 補 fallback / error handling / model migration 策略

### 9. 廣告策略看起來還在調整中，程式裡同時留下了「每次都插」與「每幾次才插一次」兩種想法

證據：

- `lib/services/storage_service.dart:75-87`
- `lib/views/divination_result_screen.dart:616-618`
- `lib/views/divination_result_screen.dart:631-636`

現況：

- 你目前的實作，確實是 AI dialog 關閉時就直接 `AdService.showInterstitialAd()`
- 如果這是你現在想要的商業策略，那它本身不是 bug
- 但同時，程式裡還保留了 `StorageService.incrementAndCheckInterstitial()`
- 這會讓後續維護的人以為系統原本有設計「每 3 次才插一次」的節流邏輯
- 換句話說，問題不一定是你現在「每次關閉都插廣告」這件事，而是專案裡同時殘留兩套不同方向，讓規則不夠清楚

白話解釋：

- 這比較像你店裡門口同時貼了兩張公告：
- 一張寫「每位客人都要排隊驗票」
- 另一張寫「每三位客人才抽查一次」
- 實際上店員現在是每個人都驗
- 那問題不是店員一定做錯，而是制度文件和現場做法沒有完全統一
- 之後如果你自己或別人回來改這段，很容易誤判原本規則

你現在的兩個真實考量我幫你拆開講：

- 你想賺錢：這很合理，廣告不是不能放
- 你也擔心太煩：這也對，因為「每次關閉都插頁」通常是最容易讓人煩的一種做法

如果以使用者體感來看：

- 獎勵式廣告：接受度最高，因為使用者知道自己是「換取額度」
- 插頁廣告：最容易有收入，但也最容易讓人反感
- 「每次關閉結果都插」：短期可能提高展示量，但也最容易造成留存下降、評價下降、甚至讓人乾脆不用 AI


如果你現在比較想採用一個保守、務實、不容易讓人暴怒的版本，我建議先這樣：

- 免費模式：只用 rewarded ad 換 AI 次數
- BYOK：先免費保留，但不要特別高調推
- interstitial 不要放在「AI 結果關閉」這個點
- interstitial 可以改成：
- 每日第一次進入紀錄頁 / 學習頁時偶爾出一次
- 或每個 session 最多一次
- 等你有更多使用者行為數據，再決定要不要把 BYOK 改成付費解鎖

「使用者行為數據」要在哪裡看？

- 先說結論：**以目前這個 repo 來看，還沒有接任何 app 內分析 SDK，所以現在其實沒有地方看「使用者在 app 裡按了什麼、在哪裡流失、看了多少廣告、多少人用 BYOK」這種細節數據。**
- 目前你之後上架後，最先能看到的是兩種平台後台數據：
- Apple 的 App Store Connect App Analytics：偏商店與下載表現，例如被多少人看到、下載、轉換
- Google Play Console 的 Dashboard / Statistics / Android vitals：偏安裝、留存、評分、崩潰、技術品質
- 但這兩種後台都不會自動告訴你像「多少人按了起卦」「多少人打到 AI 結果頁就離開」「多少人看完 rewarded ad 之後繼續用」這種 app 內行為路徑

如果你要看真正的 app 內使用者行為，通常要另外接 analytics 工具。

對這個專案最務實的第一步，我會建議：

- 先接 Firebase Analytics
- 理由是 Flutter 整合成熟、文件完整，而且可以先從基本事件開始，不一定要一次做很重

你最值得追的事件，大概是這幾個：

- `start_divination`
- `divination_result_view`
- `ai_interpretation_requested`
- `rewarded_ad_shown`
- `rewarded_ad_completed`
- `interstitial_shown`
- `record_saved`
- `record_exported`
- `record_imported`
- `byok_key_saved`
- `byok_mode_used`

如果你有了這些數據，你之後才真的能回答這些商業問題：

- rewarded ad 完成率高不高
- interstitial 是不是讓人直接離開
- 有多少人真的去設定 BYOK
- BYOK 使用者是不是比一般使用者更常回來
- 哪個頁面最容易讓人流失
- 紀錄 / 學習頁到底有沒有人看

所以「等你有更多使用者行為數據」這句話，翻成白話就是：

- 先不要只靠感覺決定廣告策略
- 先讓 app 能記錄幾個關鍵事件
- 上線後看 2 到 4 週，再決定 BYOK 要不要付費、interstitial 要不要降頻

風險：

- 若你最後其實不想每次都插，現在的實作會比你理想中更煩
- 若你最後就是要每次都插，那目前的主要問題是 dead code 與規則描述不清
- 使用者體驗若太刺激，會影響留存與評價
- BYOK 文案寫「部分轉場期間仍會保留插頁式廣告」，但實際邏輯更接近「每次關閉都可能插」

建議做法：

- 先把商業規則講清楚，只留一套
- 如果你要「每次關閉都插」，那就把 `incrementAndCheckInterstitial()` 這類沒用到的節流邏輯移掉
- 如果你要「不要那麼煩」，就把節流邏輯真的接回去
- 將文案、設定與實作三者統一

### 11. `.env` 被打包進 app asset，與安全文件敘述方向衝突

證據：

- `pubspec.yaml:95-99`
- `lib/main.dart:17`
- `Security.md:14-18`

現況：

- `.env` 被列入 Flutter assets
- App 啟動時直接 `dotenv.load(fileName: ".env")`
- 目前 `.env` 內容雖然只有 placeholder API key 與 worker URL，但這種設計會鼓勵把敏感設定打包進客戶端

白話解釋：

- `.env` 不是保險箱，它比較像「方便整理設定的小筆記本」
- 在後端專案裡，`.env` 常常放在伺服器上，不跟著程式一起發出去，所以可以放敏感資料
- 但你現在這個是 Flutter app，而且 `pubspec.yaml` 明確把 `.env` 放進 assets
- 只要被放進 assets，意思就是：這份檔案會跟著 app 一起被打包進 APK / IPA
- 換句話說，它不再只存在你的電腦，而是會隨 app 一起送到使用者裝置上
- 使用者平常不一定「直接看到」它，但 technically 它已經在 app 安裝包裡了
- 所以 `.env` 在手機 app / 前端 app 裡，從來都不是「無敵隱藏」
- 它只能算是方便管理設定，不能拿來保護真正的秘密

再講得更直白一點：

- 後端的 `.env`：比較可以放秘密
- 前端 / 手機 app 的 `.env`：不能當秘密倉庫

以你現在這個專案來說：

- 如果 `.env` 只是放 `WORKER_URL` 這種公開也沒關係的設定，問題不大
- 但如果哪天把正式 Gemini API key、廣告 secret、任何敏感 token 放進去，就等於跟著 app 一起發出去了
- 所以我不是在說「你現在已經外洩重大機密」，而是在說「這個結構很容易讓未來的人誤以為可以安全地把秘密放進 `.env`」

風險：

- 現在雖未看到真的把正式 Gemini key 打進去，但架構已經把風險打開了
- 與「官方 Gemini key 絕對不會被打包進 App」的安全文件說法方向衝突

建議做法：

- 客戶端只保留不敏感配置
- 敏感金鑰只放 server secret / device secure storage
- 若仍需要 client-side build config，改成區分 dev / release 並限制內容
- 最簡單的原則是：
- `WORKER_URL` 這種可公開設定，可以留在 client config
- 真正的 secret，不要放在 app 會打包出去的 `.env`

### 13. 有些 `StatefulWidget` 持有 controller 卻沒有 `dispose`

證據：

- `lib/views/divination_screen.dart:18-30` 有多個 `TextEditingController`
- `lib/views/settings_screen.dart:12-16` 有 `TextEditingController`
- 但兩個類別都沒有 `dispose()`

風險：

- 這不是馬上爆炸的 bug，但屬於基本資源釋放問題
- 長期來看會讓畫面生命週期管理變得不乾淨

白話解釋：

- `controller` 可以把它理解成「畫面元件背後的小助手」
- 例如輸入框要記住你打了什麼字，就常會配一個 `TextEditingController`
- 但當這個畫面被關掉、離開後，這個小助手也應該一起下班
- `dispose()` 就是告訴 Flutter：「這個畫面不用了，相關資源可以收掉了」
- 如果沒做，短期可能看不出問題，但長期來說就像畫面關掉了，卻還留一些小助手沒收乾淨
- 這通常不會讓 app 立刻壞掉，但屬於程式衛生習慣不好，累積久了比較容易出現奇怪問題

建議做法：

- 補 `dispose()`，統一釋放 controller
- 順便整理 widget 生命週期邏輯
- 這個項目優先級沒有前面那些上架阻塞那麼高，但屬於應該順手補齊的基本功

## 五、程式碼整理建議

### 23. 重新切分層次：View / Flow / Domain / Infra

建議方向：

- `views/` 只保留畫面與互動
- `controllers` 或 `usecases` 管流程，例如起卦、儲存紀錄、請求 AI、廣告獎勵
- `services/` 保留純邏輯或外部服務接入
- `repositories/` 只處理資料讀寫

理由：

- 目前很多畫面檔直接碰 repository、service、ad flow、navigation，責任太重

### 24. 先抽出共用 domain 函式

優先抽：

- 卦象 ID 計算
- 爻位名稱轉換
- 變卦繪製轉換
- AI prompt 組裝器

理由：

- 這些邏輯重複多，而且是核心規則

### 25. 為流程建立明確狀態機，而不是在畫面中疊加布林值

目前例子：

- `DivinationScreen` 用 `_isAnimating`、`_isCoinRolling`、`_selectedMethod`、`_coinLines`
- `_AIDialogContent` 用 `_isLoadingAd`、`FutureBuilder`、credits refresh

問題：

- 小案子還能維持，但一旦增加 loading / retry / cancel / paywall / offline 分支，很快會失控

建議做法：

- 用 Riverpod Notifier / AsyncNotifier 建立可測試的狀態模型

### 26. 把產品規則寫成常數或設定，而不是散落 magic number

目前例子：

- 每次看廣告給 3 點：`lib/views/divination_result_screen.dart:479-483`
- 每日基礎點數 3：`lib/services/storage_service.dart:41`, `55-58`
- 插頁廣告每 3 次一次：`lib/services/storage_service.dart:78-86`

問題：

- 規則散落在多處，不易統一修改

建議做法：

- 建立 `app_config.dart` 或 monetization config

## 六、離送審還差哪些「repo 外」工作

以下不一定能從 repo 內直接驗證，但上架前一定要補：

### 目前帳號狀態補充（依最新提供資訊）

- Apple Developer 帳號：還沒有
- Google Play Console 帳號：還沒有
- AdMob 帳號：已經有，而且基本設定已完成

這代表什麼：

- 你現在的 monetization 準備，反而比「商店上架準備」更前面
- 換句話說，你已經先把「怎麼放廣告」準備了一部分
- 但「怎麼正式發佈到商店、怎麼看商店後台數據、怎麼進入正式審查」這一層還沒開始

白話解釋：

- AdMob 像是你已經準備好收銀機
- 但 Apple Developer / Play Console 才像是你要開店時需要的店面執照與商場櫃位
- 只有收銀機，不代表你已經能開店
- 所以你現在不是不能做 app，而是還不能走到真正的商店上架流程

對實際進度的影響：

- iOS：沒有 Apple Developer 帳號，就還不能走正式 App Store / TestFlight 發佈流程，也不會有 App Store Connect 後台數據
- Android：沒有 Google Play Console 帳號，就還不能上 Google Play，也不會有 Play Console 的商店後台、Android vitals、商店流量數據
- AdMob 雖然已就緒，但要真的驗證廣告收入效果，還是需要進入更接近真實發佈的環境

我對你現在的順序建議：

- 如果你近期最想先驗證「能不能上架 + 能不能賺到廣告收入」，優先順序應該是：
- 先建立 Google Play Console
- 先把 Android 發佈流程打通
- 補最基本的 analytics / 事件追蹤
- 跑一輪測試版，先觀察廣告完成率與流失
- Apple Developer 則放在你準備開始 iOS TestFlight / App Store 發佈前處理

原因很簡單：

- 你現在已經有 AdMob
- Android 上廣告驗證與測試通常會比 iOS 先跑起來
- 而且一旦進到 Play Console，你才能開始累積比較接近真實市場的數據

### App Store 需要補的

- App Store Connect 的 privacy policy 公開 URL
- App Privacy Nutrition Labels
- Support URL 與聯絡方式
- 最終上架文案、截圖、預覽影片、關鍵字、分類、年齡分級
- 實機測試紀錄，至少覆蓋最新 iOS 與常見螢幕尺寸

### Google Play 需要補的

- 正式 release signing / Play App Signing
- Data safety form
- 公開 privacy policy URL
- 商店 listing、截圖、聯絡資訊、年齡分級
- 關於廣告與資料處理的準確揭露
- 目標 API level 與實際 build artifact 驗證

說明：

- 依 Google Play 官方規則，自 2025-08-31 起，新 app 與更新需 target Android 15 (API 35) 或更高；這個專案使用的是 `flutter.targetSdkVersion`，從 repo 看不出最終實際值，需要在正式 build artifact 再確認

   const hashBuffer = await crypto.subtle.digest('SHA-256', data);

### Q2 結論：插頁廣告改為有節制版本

以下是四個適合這個專案的節制策略模板，你看了之後選一個：

#### 模板 A：冷卻時間制（推薦）

規則：
- 兩次插頁廣告之間至少間隔 **10 分鐘**
- 不管使用者中間做了什麼（看 AI、關 AI、切頁面），10 分鐘內最多只出現 1 次

Future<bool> canShowInterstitial() async {
  final lastTime = _prefs.getInt(_kLastInterstitialTime) ?? 0;
  final now = DateTime.now().millisecondsSinceEpoch;
  return (now - lastTime) >= 600000; // 10 分鐘 = 600000ms
}

Future<void> recordInterstitialShown() async {
  await _prefs.setInt(_kLastInterstitialTime, DateTime.now().millisecondsSinceEpoch);
}

- 超過就不再顯示，隔天重置

static const _kDailyInterstitialCount = 'daily_interstitial_count';

  final savedDate = _prefs.getString(_kDailyInterstitialDate) ?? '';
  if (savedDate != todayStr) {

    await _prefs.setInt(_kDailyInterstitialCount, 0);
  }
  final count = _prefs.getInt(_kDailyInterstitialCount) ?? 0;
  return count < 3;
}

Future<void> recordInterstitialShown() async {
  final count = _prefs.getInt(_kDailyInterstitialCount) ?? 0;

### Q4 結論：Google AdMob Server-Side Verification (SSV) 實作說明

以下是完整的 SSV 實作步驟，你看完決定要不要做：

#### 什麼是 SSV？

白話版：使用者看完 rewarded ad 後，Google 不是只告訴你的 app「他看完了」，而是 Google 會**直接打一個 callback 到你的 server（Worker）**讓你的 server 去驗證「這個獎勵是不是真的來自 Google」。

#### 要做哪些事？

**第 1 步：AdMob 後台設定 SSV callback URL**（5 分鐘）
- 進入 AdMob → 你的 app → 廣告單元設定
- 設定 Server-Side Verification callback URL 為你的 Worker 端點，例如：
  `https://iching-gemini-proxy.lumus1128.workers.dev/verify-reward`
  
**第 2 步：Worker 增加驗證端點**（約 1-2 小時）
- Worker 新增一個 GET 路由 `/verify-reward`
- Google 會帶以下 query params 打過來：
  - `ad_network`, `ad_unit`, `custom_data`, `reward_amount`, `reward_item`, `timestamp`, `transaction_id`, `user_id`, `signature`, `key_id`
- Worker 需要：
  1. 用 Google 公開的 AdMob 公鑰驗證 `signature`（以確認這個 callback 真的來自 Google）
  2. 驗證 `transaction_id` 沒有被重複使用（防重放攻擊）
  3. 驗證通過後，把 `custom_data`（你可以帶使用者的 anonymous ID）記錄為「已驗證獎勵」

**第 3 步：Flutter 端在看完 rewarded ad 後傳 custom_data**（30 分鐘）
- 在呼叫 `RewardedAd.load` 時，設定 `ServerSideVerificationOptions`：
  ```dart
  ad.setServerSideVerificationOptions(
    ServerSideVerificationOptions(customData: '一個唯一的 session token'),
  );
  ```
- 看完廣告後，改成先去你的 Worker 確認這個 token 有被 Google callback 驗證過，再給點數

**第 4 步：Worker 端增加一個「查詢獎勵是否已驗證」的 API**（30 分鐘）
- Flutter app 看完廣告後，呼叫 Worker 的另一個端點確認該 token 是否已被 Google SSV callback 確認

#### 總工作量評估

| 步驟 | 預估時間 | 難度 |
|------|---------|------|
| AdMob 後台設定 | 5 分鐘 | 低 |
| Worker 驗證端點 | 1-2 小時 | 中（需要處理 ECDSA 簽名驗證） |
| Flutter 端改動 | 30 分鐘 | 低 |
| Worker 查詢端點 | 30 分鐘 | 低 |
| 測試 | 1 小時 | 中 |
| **總計** | **約 3-4 小時** | 中 |

#### 不做 SSV 的風險

- 如果有人反編譯你的 app，直接呼叫你的 Worker API（帶一個假的 adToken），就能無限拿到 AI 解卦額度而不看廣告
- 以你現在的 app 規模（還沒上架），這個風險短期內很低
- 但如果你的 app 之後有一定使用量，這個漏洞會被人發現

#### 我的建議

以你目前的階段：**先不做 SSV，先上架**。理由：
1. 你的 app 還沒上架，使用者量為 0，被攻擊的機率極低
2. SSV 的 3-4 小時工作量不算大，但如果在上架前做，增加了測試複雜度
3. 等你上架後、有了第一批使用者且發現有人濫用時，再補 SSV 即可
4. 到時候補 SSV 不影響現有功能，是純後端改動
