# 籌策模擬起卦設計

日期：2026-05-06

狀態：草案待審

## 背景

ChangeLog 目前在進階起卦方式中已有「籌策」選項，但實際行為是讓使用者手動輸入六個 `6/7/8/9` 爻值。這對「已經有六爻結果」的使用者有用，卻不符合多數人對「籌策」的期待：用蓍草或籌策進行四營十八變，逐步得出六爻。

本規格將「籌策」重新定義為真正的籌策模擬流程。App 以數學與程式模擬實體籌策操作，讓使用者不必準備實物，也能用接近傳統方法的方式完成起卦。

原本的手動輸入能力不在本規格內處理。它會移到獨立的「手動卦象輸入」規格，涵蓋兩類情境：使用者已用實體籌策取得六爻結果，以及使用者透過象、數、物候等外部線索得到卦象與變爻。

金錢卦動畫與易經查詢頁轉場問題也不在本規格內處理，另由「占卦與易經查詢視覺優化」規格承接。

## 目標

- 將進階起卦中的「籌策」改為真正模擬籌策起卦。
- 使用者按一次即可自動完成六爻，但畫面逐爻演示過程。
- 演算法逐步模擬分策、掛一、揲四、歸奇，而不是直接抽取最後爻值。
- 讓結果可追溯：使用者可展開查看每一爻、每一變的數值明細。
- 預設保存完整過程，但允許使用者關閉，僅保存結果。
- 保存資料使用結構化 JSON，而不是保存畫面文字或程式碼。

## 非目標

- 不設計手動六爻輸入、外部得卦、卦象直接輸入或變爻位置輸入。
- 不重做金錢卦流程或動畫。
- 不重做易經查詢頁、六十四卦卡片或詳情頁轉場。
- 不在主流程展示完整機率公式或學術推導。
- 不提供重新模擬來補回未保存的過程，避免讓使用者誤以為可以重建原本那次起卦。

## 使用者流程

進階起卦方式中的「籌策」入口改為真正的籌策模擬。

使用者進入籌策後，畫面提供：

- 主要按鈕：`模擬籌策`
- 偏好開關：`保存完整過程`
- 簡短提示：說明籌策會逐步模擬傳統起卦過程，完整明細可於結果中展開查看

使用者按下 `模擬籌策` 時，App 立即完成同一次完整演算，產生六爻與十八變明細。動畫只是播放這份已生成的過程。若使用者按下 `略過動畫`，畫面直接跳到同一組結果，不重新演算。

預設流程是一次按下後自動跑完整六爻。使用者不需要每一爻按一次，也不需要逐步點完十八變。未來可再考慮慢速模式，但第一版不納入必要範圍。

## 動畫設計

動畫採「儀式 + 可讀狀態」方向。

畫面主體用簡化籌策分合動畫，讓使用者看見籌策正在被分成左右兩堆、取出一策、揲四、收集餘策。旁邊或下方同步累積六爻結果。每完成一爻，就顯示該爻的：

- 爻位：初、二、三、四、五、上
- 數值：`6/7/8/9`
- 爻性：老陰、少陽、少陰、老陽
- 是否變爻

動畫文字以傳統術語為主，白話輔助：

- `分二`：將籌策分為左右兩堆
- `掛一`：取出一策象徵三才
- `揲四`：左右兩堆以四策為一組
- `歸奇`：收集餘策，進入下一變

動畫完成後進入結果畫面，顯示本卦、之卦、互卦、變爻與朱熹解卦法則建議。籌策過程明細以可展開區塊呈現，不壓迫主流程。

## 演算法

籌策模擬採逐步分策，不直接依結果機率抽 `6/7/8/9`。

每一爻從 `49` 策開始，符合「大衍之數五十，其用四十有九」。每一爻經過三變，每一變流程如下：

1. `分二`：將目前策數隨機分為左右兩堆，每堆至少一策。
2. `掛一`：從右堆取出一策。
3. `揲四`：左右兩堆各自除以四取餘數。若餘數為 `0`，以 `4` 記。
4. `歸奇`：扣除掛一、左餘、右餘。
5. 將剩餘策數帶入下一變。

三變完成後，將剩餘策數除以 `4`，得到一爻：

- `6`：老陰，變爻
- `7`：少陽，靜爻
- `8`：少陰，靜爻
- `9`：老陽，變爻

六爻由下而上生成，與現有 App 中 `lines[0]` 為初爻的資料方向一致。

## 明細資料

籌策結果的完整明細以結構化資料保存。畫面文字由 App 根據結構化資料產生，不把長篇說明文字直接存進資料庫。

每一筆籌策明細包含：

- 起卦方式類型
- 明細 schema 版本
- 每一爻的三變資料

六爻結果本身不在 `methodDetailJson` 內另存一份。它以既有共用欄位 `rawHexagramNumbersStr` 作為唯一來源。籌策明細只保存「如何推得這些爻」的過程資料；若需要顯示每一爻的 `6/7/8/9`，可由該爻第三變的 `after / 4` 推得，並與 `rawHexagramNumbersStr[position - 1]` 做一致性檢查。

每一變至少保存：

- `changeIndex`：本爻中的第幾變，值為 `1/2/3`
- `before`：本變開始前策數
- `left`：分二後左堆策數
- `right`：分二後右堆策數，尚未掛一前
- `hang`：掛一策數，固定為 `1`
- `leftRemainder`：左堆揲四餘策
- `rightRemainder`：右堆扣除掛一後揲四餘策
- `removed`：本變扣除總策數
- `after`：本變結束後剩餘策數

概念資料格式如下。實際每一爻應有三筆 `changes`；此處為縮短篇幅，只示範第一變。

```json
{
  "type": "yarrow",
  "version": 1,
  "lines": [
    {
      "position": 1,
      "changes": [
        {
          "changeIndex": 1,
          "before": 49,
          "left": 24,
          "right": 25,
          "hang": 1,
          "leftRemainder": 4,
          "rightRemainder": 4,
          "removed": 9,
          "after": 40
        }
      ]
    }
  ]
}
```

明細可用於：

- 回放或展開查看原始過程
- 驗證 `before - removed = after`
- 驗證三變後 `after / 4` 等於 `rawHexagramNumbersStr` 中對應位置的爻值
- 未來調整顯示文案或支援多語系

## 機率說明

主流程不展示完整機率公式，也不把機率差異當成主要賣點。

若使用者展開明細或進入說明，可用白話簡短說明：籌策模擬是逐步分策得出結果，老陰、少陽、少陰、老陽的分布與金錢卦不同。因此它不是金錢卦動畫換皮，也不是直接用相同機率抽結果。

這段說明不需要在第一版列出完整分數或學術推導。

## 紀錄保存

占卦紀錄新增通用可選欄位 `methodDetailJson`，而不是籌策專用欄位。

籌策紀錄保存：

- `method`：顯示為 `籌策`，內部可使用穩定 code，例如 `yarrow`
- `rawHexagramNumbersStr`：共用的六爻結果欄位，例如 `[6, 8, 7, 9, 8, 7]`。這是本筆紀錄的唯一六爻結果來源，與數字占、金錢卦等其他方式共用。
- `methodDetailJson`：若使用者保存完整過程，存籌策明細 JSON；若使用者關閉保存完整過程，存 `null`

採用通用欄位的原因是未來其他起卦方式也可能需要方法明細。例如金錢卦可能保存每一爻三枚銅錢結果，手動卦象輸入可能保存外部得卦來源與輸入方式。紀錄系統只需要知道「某些起卦方式可以附帶方法明細」，不需要為每個方法建立專用欄位。

`methodDetailJson` 不應成為第二份六爻結果來源。它可以讓 App 重新生成籌策過程說明，但結果頁、紀錄列表、卦象計算與 AI 解卦都應讀取 `rawHexagramNumbersStr`。

## 保存完整過程偏好

籌策畫面的 `模擬籌策` 按鈕附近提供 `保存完整過程` 開關。

行為：

- 預設開啟。
- 使用者改變後，App 記住上次選擇。
- 下次進入籌策模擬時沿用上次狀態。
- 開關只影響籌策模擬，不影響數字占、金錢卦或手動卦象輸入。
- 開啟時，紀錄保存六爻結果與完整十八變明細。
- 關閉時，當下仍完整模擬並播放動畫，但紀錄只保存結果，不保存十八變明細。
- 起卦完成後不再詢問是否補保存。

此偏好適合放在 SharedPreferences，由現有 `StorageService` 管理。

## 紀錄詳情顯示

紀錄詳情頁用低調標籤表示籌策紀錄的完整度：

- `有過程`：此紀錄保存完整籌策過程，可展開十八變明細。
- `僅結果`：此紀錄只保存卦象結果，未保存籌策過程。

若 `methodDetailJson` 為空，不顯示錯誤，也不提供重新模擬按鈕。明細區可用小字說明：「此紀錄只保存卦象結果，未保存籌策過程。」

## 驗收標準

- 進階起卦中的「籌策」不再是手動輸入六爻，而是真正的籌策模擬。
- 使用者按一次 `模擬籌策` 即可產生完整六爻。
- 動畫逐爻播放已生成的同一次演算過程。
- `略過動畫` 不會重新演算，只會跳到同一組結果。
- 演算法逐步模擬分二、掛一、揲四、歸奇。
- 六爻由下而上生成，並與現有卦象計算邏輯相容。
- 完成後可展開查看每一爻、每一變的結構化明細。
- 主流程不展示完整機率公式，機率差異只在說明或明細中白話呈現。
- `保存完整過程` 開關預設開啟，並記住使用者上次選擇。
- 開關開啟時，紀錄保存 `methodDetailJson`；關閉時，紀錄只保存結果。
- 紀錄詳情頁以 `有過程` / `僅結果` 標籤低調顯示保存完整度。
- 原本手動輸入六爻不在本規格內實作，另由手動卦象輸入規格承接。
